Facade Pattern:
Provides a unified interface to a set of interfaces in a subsystem.
Facade defines a higher-level interface that makes the subsystem easier to use.


User want's to watch movie in DVD. In order to watch movie user has to do some
set of operation sequentially rather than just one click to "watchMovie" and the same
is true for "endMovie". So not user friendly. What user wants is that user will click on one button and set the movie user wants to watch and while setting off the system user wants to click one button "endMovie"

Question & Condition:

  1. In future we can add any subsystem in order to watch movie. We should be able to add
    any subsystem, but client should not bother about this new subsystem, user still want:
    "I will click "watchMovie" and "endMovie"
  2. We need to decouples a client from a complex subsystem.
Solutions & Design principles:
  1. We will use Facade pattern
  2. We can implement more than one facade for a subsystem
So Object Oriented Principles:
  • Encapsulate what varies
  • Favor composition over inheritance (HAS-A can be better than IS-A)
  • Program to interfaces, not to implementations
  • Strive for loosely coupled designs between objects that interact
  • Class should be open for extension but closed for modification
  • Depend on abstraction. Do not depend on concretion
  • Only talk to your fiends (Facade Pattern..will come soon)
