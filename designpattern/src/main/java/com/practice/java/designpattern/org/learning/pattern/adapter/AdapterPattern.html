Adapter Pattern:
Converts one interface to another
It converts the interface of class into another interface the clients except.
Adapter lets classes work together that couldn't otherwise because of incompatible interfaces.


You've got an existing software system that you need to work a new vendor class library
into, but the new vendor designed their interfaces differently than the last vendor: Hope you remember the DuckSimulator (Strategy pattern): It has two method:
  1. quack();
  2. fly();
Now it's time to meet the newest fowl on the block: Turkey:
  1. gobble(); (don't quack)
  2. fly(); (They can fly small distance)

Question & Condition:

  1. You don't want to solve the problem by changing your existing code (and you can't
    change the vendor's code)
  2. "An un-coupled client is a happy client"- Client only knows Target interface
Solutions & Design principles:
  1. We will use Adapter pattern:
    We will provide a Adapter class in-between Duck and Turkey (Adaptee) and
    Adapter class is responsible to translate the request into new request, so client
    did not to know what is happening underline.
  2. Two kinds of Adapter Pattern:
    1. Object Adapters:
      In Object adapter Adapter use composition to adopt a Adaptee
    2. Class Adapters:
      In Class adapter Adapter do not use composition to adopt a Adaptee.
      The Adapter now subclasses the Adaptee and the Target classes.
So Object Oriented Principles:
  • Encapsulate what varies
  • Favor composition over inheritance (HAS-A can be better than IS-A)
  • Program to interfaces, not to implementations
  • Strive for loosely coupled designs between objects that interact
  • Class should be open for extension but closed for modification
  • Depend on abstraction. Do not depend on concretion
  • Only talk to your fiends (Facade Pattern..will come soon)
